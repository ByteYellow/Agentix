The two **plugin axes** ([Namespace](/namespace) and
[Deployment](/deployment)) exist because they cross the hostâ†”sandbox
boundary. Everything else lives on the host as plain Python â€” you
extend it by importing and calling, not by `pip install`. This guide
walks each host-side hook.

## Trace sinks â€” fan-out observability

Every namespace's `agentix.trace.emit(kind, payload)` call fans out to
every registered sink. Sinks register once at runtime startup via an
**installer** â€” a callable the framework invokes during lifespan:

```python title="my_otel_sink/__init__.py"
from opentelemetry import trace as otel
from agentix.trace import register_sink

def install() -> None:
    tracer = otel.get_tracer("agentix")

    def sink(kind, payload, call_id, source):
        with tracer.start_as_current_span(f"agentix.{kind}") as span:
            span.set_attribute("agentix.call_id", call_id or "")
            span.set_attribute("agentix.source", source or "")
            for k, v in payload.items():
                span.set_attribute(f"agentix.payload.{k}", v)

    register_sink(sink)
```

Call `install()` from your own startup code:

```python
import my_otel_sink
my_otel_sink.install()
```

Sink errors are caught and logged â€” tracing never breaks a rollout.
But sinks that **block** will slow tracing for the entire runtime;
offload heavy work (network I/O, file flushes) to a background thread
or async task inside the sink body.

<Tip>
  **Why isn't this an entry-point axis?** Trace sinks run host-side,
  not in the sandbox. Plugin discovery via `pip install` is for
  things that cross the process boundary. A host-side hook is one
  function call â€” `import` and `register_sink(my_sink)` is enough.
  No framework dispatch needed.
</Tip>

## Wire patterns â€” three built-ins, not extensible

Three call shapes ship in `agentix.wire`:

| Pattern         | Class             | Matches signatures                                                |
| --------------- | ----------------- | ----------------------------------------------------------------- |
| Unary           | `UnaryPattern`    | plain return (`-> T`)                                             |
| Server-stream   | `StreamPattern`   | `-> AsyncIterator[T]`                                             |
| Bidirectional   | `BidiPattern`     | `AsyncIterator[U]` first param + `-> AsyncIterator[V]`            |

Selection is at `Dispatcher.bind` time â€” the framework inspects each
namespace method's signature and picks the most specific match. There
is **no `register_pattern` hook**. If a fourth call shape ever becomes
necessary, add it as a `WirePattern` subclass in `agentix/wire.py` and
update the in-tree registry. Wire framing is core to the protocol; it
doesn't belong in third-party wheels.

## Spec resolvers â€” internal ordered list

`agentix build <spec>` accepts four spec shapes today: a local path
(`./primitives/bash`), a docker image ref
(`docker.io/me/agent:1.0`), a short name resolved against the repo
(`bash`), and a PyPI dist (`agentix-bash`). The resolution chain lives
in `agentix/cli/_resolve.py`:

```python
RESOLVERS = [
    PathResolver(priority=100),
    ImageResolver(priority=90),
    LocalRepoResolver(priority=50),
    PypiResolver(priority=10),
]
```

The CLI walks the list in priority order; the first resolver whose
`resolve()` returns non-`None` wins. **Add a new spec shape by editing
that file** â€” extending the CLI's input grammar is a framework concern,
not a per-wheel one.

## A new `agentix <verb>` CLI

The four built-in subcommands (`build`, `install`, `deploy`, `check`)
are hardcoded in `agentix/cli/__init__.py`. The dispatcher is *not* a
plugin surface. Third parties that want a new top-level verb should
ship a separate `console_scripts` binary in their wheel:

```toml title="pyproject.toml"
[project.scripts]
agentix-yourcmd = "your_package:main"
```

Users invoke it directly: `agentix-yourcmd â€¦`. The convention is
`agentix-<verb>` so it sorts alphabetically next to the framework
verbs in shell completion.

<Tip>
  **Why not a plugin?** The CLI dispatcher is a thin argparse handler
  with no per-subcommand state. Loading every wheel's subcommands at
  every CLI invocation would slow down `agentix --help` and turn a
  static argparse tree into a dynamic registry. A `console_scripts`
  binary is the same UX with none of that cost.
</Tip>

## Summary

| Surface                        | Mechanism                          | Where to add it                              |
| ------------------------------ | ---------------------------------- | -------------------------------------------- |
| Trace sinks                    | `register_sink(fn)` at startup     | Your own host-side startup code              |
| Wire patterns                  | In-tree `WirePattern` subclass     | `agentix/wire.py` (framework PR)             |
| Spec resolvers                 | In-tree ordered list               | `agentix/cli/_resolve.py` (framework PR)     |
| A new `agentix <verb>`         | Separate `console_scripts` binary  | Your own wheel â€” `agentix-<verb>` convention |

If you find yourself wanting to ship a new wire pattern or spec
resolver as a plugin, file an issue first â€” that's a framework-level
design decision, not a wheel-level one.
