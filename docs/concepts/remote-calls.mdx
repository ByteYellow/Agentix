---
title: Remote calls
description: How Agentix turns an imported Python function into a sandboxed call.
---

A remote call starts with a real function object, not a string route.

```python
from app import run

result = await client.remote(run, input="hello")
```

Agentix reads exactly two attributes from `run`:

```python
target = f"{run.__module__}::{run.__qualname__}"
```

The runtime worker receives that target, imports the module inside the
sandbox, resolves the function, unpickles the arguments, and calls it.

## Why Function Objects

Passing the function object keeps the host and sandbox API aligned:

- Your editor sees kwargs and return types from the local signature.
- Refactors stay visible to Python tooling instead of hiding in RPC
  strings.
- Agents, tools, and scorers all use the same call path.

The host rejects callables that cannot be encoded as an import path
(lambdas, bound methods, partials, local closures). Put remote code
behind an importable top-level function instead.

## Payload Shape

This call:

```python
from app import run

result = await client.remote(run, input="hello")
```

becomes this wire payload:

```python
{
    "callable": "app::run",
    "arguments": pickle.dumps(((), {"input": "hello"})),
    "call_id": "uuid-hex",
}
```

`RuntimeClient.remote()` generates a fresh `call_id` per call. The
runtime uses it to correlate `call:result` / `call:error` responses and
to support cancellation.

Remote calls use Socket.IO events on the `/` namespace. HTTP is only
used for `/health`.

## Example

```python title="app.py"
async def run(input: str) -> dict[str, str]:
    return {"output": input.upper()}
```

```python
from app import run

result = await client.remote(run, input="hello")
```

Both sync and async functions are valid targets. The worker awaits the
return value only when it is awaitable.

## Side Channels

Trace, log, and plugin traffic share the same Socket.IO connection but
ride their own namespaces, separate from `c.remote()`:

| Namespace | Direction | Purpose |
| --- | --- | --- |
| `/` | host ↔ sandbox | `c.remote()` |
| `/trace` | sandbox → host | span lifecycle (auto-registered) |
| `/log` | sandbox → host | stdlib logging records (auto-registered) |
| `/<plugin>` | both | plugin-defined events via `agentix.sio` |

Register a host-side handler before connecting:

```python
client = RuntimeClient(url)
client.register_namespace(MyHostNamespace())
async with client as c:
    await c.remote(my_sandbox_fn, ...)
```

## Errors and Cancellation

Agentix keeps errors in-band. A failed call emits `call:error` with a
structured payload; the client raises `RemoteCallError` (or
`asyncio.CancelledError` when `error.cancelled=True`).

When the client disconnects or the caller's task is cancelled before a
result arrives, the client emits `cancel` for the in-flight `call_id`.

Per-call timeouts are caller-owned — wrap `await client.remote(...)` in
`asyncio.wait_for(...)` when you need a deadline.
